## 一、项目背景与目标

### 1.1 业务背景

某连锁餐饮品牌在全国多个市场运营外卖业务，覆盖**自有小程序配送**和**第三方外卖平台**两个渠道，总门店数约 **500 家**，由数十位区域管理人员分层负责。

在项目实施前，团队面临三个核心痛点：

| 痛点 | 具体表现 |
|------|----------|
| **数据分散** | 两个平台数据各自独立，无法交叉分析；门店架构信息散落在多个 Excel 文件中，没有统一的"门店-管理者"映射 |
| **发现问题慢** | 依赖人工逐日导出报表、手动比对，一个门店的出餐异常从发生到被管理者感知通常滞后 2-3 天 |
| **决策缺依据** | 管理者只能看到自己辖区的零星数据，无法横向对比、不知问题严重程度在全盘中的位置 |

### 1.2 项目目标

构建一个**单页面的综合运营看板**，实现三个核心能力：

1. **一眼看全貌** — 大盘数据、渠道表现、管理排名在同一页面呈现，无需切换
2. **一层层下钻** — 从市场 → 大区 → 区经理 → 单店，逐级定位问题
3. **自动预警** — 异常指标自动标红、自动生成预警清单，把"人找问题"变成"问题找人"

### 1.3 项目范围

| 维度 | 覆盖范围 |
|------|----------|
| 平台 | 自有小程序配送 + 第三方外卖平台 |
| 组织层级 | 市场 → 品牌 → 大区经理 → 区经理 → 大店长 → 门店 |
| 时间范围 | 近 3 个月滚动数据，每日刷新 |
| 核心指标 | 出餐时效、店铺评分、体验评分三大类 |

## 三、数据来源与处理流程

### 3.1 数据源结构

三个核心数据表，每月一份独立的 Excel 文件（约 20-30MB）：

| 数据表 | 行数（单月） | 关键字段 |
|--------|-------------|----------|
| 美团架构数据 | ~500 行 | 美团ID、门店编码、门店名称、市场、品牌、城市、大区经理、区经理、大店长 |
| 美团指标数据 | ~15,000 行 | 门店维度 + 店铺分10项子维度 + 体验分9项子维度 |
| 小程序配送数据 | ~80,000 行 | 订单编码、门店名称、制作时长、订单杯数、日期 |

### 3.2 核心处理逻辑

#### 3.2.1 多月份数据合并

三个月份的数据按优先级合并：**7 月 > 6 月 > 5 月**。

- 架构数据：同名门店以最新月份的架构关系为准（应对人员调动场景）
- 指标数据：同一订单编号的记录，优先采用最新月份
- 合并键：美团按 (日期, 门店ID)，小程序按 (订单编码)

#### 3.2.2 门店架构匹配

源数据中，"门店的架构归属"和"门店的指标数据"来自不同的 Sheet，需要做匹配。采用五级匹配策略：

```
1. 门店编码精确匹配  ──→ 命中率 ~70%
2. 美团ID精确匹配    ──→ 命中率 +15%
3. 门店名称精确匹配  ──→ 命中率 +10%
4. 名称模糊匹配      ──→ 命中率 +3%
5. 未匹配记录        ──→ 跳过（约 2%，通常为新店或更名门店）
```

最终匹配率稳定在 **97-98%**。

#### 3.2.3 品牌与市场规范化

原始数据中品牌名称存在多种写法（"鸳央" / "鸳央咖啡"），市场名称也有变体（"江苏省" / "江苏" / "江苏省市场"）。管道中内置规范化映射表，统一清洗为标准值。

#### 3.2.4 内存优化

小程序配送数据单月约 8 万行，三月份合计约 24 万行。采用 `openpyxl` 流式读取（`read_only=True` + `iter_rows`），避免将整个 Excel 加载到内存，处理峰值内存控制在 **500MB 以内**。

> 📷 **[配图建议 3]**：数据处理管道流程图，展示"三个 Excel → 架构合并 → 指标聚合 → 预警生成 → 每日汇总 → data.json"六个阶段。标注每阶段输入/输出和关键参数。

### 3.3 数据去重与质量保障

| 问题场景 | 处理策略 |
|----------|----------|
| 同一门店在多月份出现 | 取最新月份数据 |
| 同一订单在多平台重复 | 按订单编码去重 |
| 门店名称变更（更名） | 通过门店编码匹配保持连续性 |
| 架构信息缺失 | 模糊匹配兜底，未匹配记录统计并报告 |
| 出餐时长异常值（>120分钟） | 保留原始值不丢弃，在热力图中以最深红色标识，便于排查 |

## 四、核心指标体系

### 4.1 指标分层

看板的指标体系按"决策层级"分为三层：

```
┌── 战略层 ──────────────────────┐
│  大盘总览：门店总数、平台占比、   │
│  出餐/评分/体验三大均值 + 环比    │
├── 管理层 ──────────────────────┤
│  排名系统：市场排名、品牌排名、   │
│  大区排名、区经理排名            │
├── 执行层 ──────────────────────┤
│  单店诊断：出餐逐时热力图、      │
│  评分六维明细、改善建议          │
└────────────────────────────────┘
```

### 4.2 三大核心指标

| 指标 | 来源平台 | 计算方式 | 预警阈值 | 业务含义 |
|------|----------|----------|----------|----------|
| **出餐时长** | 自有小程序 | 制作时长均值（分钟） | > 15 分钟 | 反映门店备货效率，直接影响顾客等待体验 |
| **店铺分** | 第三方平台 | 平台官方评分（0-100） | < 95 分 | 平台对门店的综合评价，影响排名和曝光 |
| **综合体验分** | 第三方平台 | 平台体验分（0-5） | < 4.6 | 顾客满意度综合指标，含商品/包装/服务/食安 |

### 4.3 评分下钻维度

每个核心指标下，拆分为多个可执行的子维度：

**店铺分（10 项子维度）**：高峰营业时长、优质商品率、不接单率、差评回复率、商家评分、菜单丰富度、装修丰富度、服务功能丰富度、出餐上报率、基础营业时长

**体验分（9 项子维度）**：商品满意度、包装满意度、复购率、消息回复率、服务负反馈率、食安负反馈率、商品质量分、服务体验分、出餐完成上报率

> 📷 **[配图建议 4]**：指标金字塔图，三层分别标注指标名称、预警阈值和业务含义。底层的子维度用进度条形式示意。

## 五、页面结构与功能模块

### 5.1 页面全景

看板为单页面无限滚动布局，自上而下共 **11 个功能区**：

| 序号 | 功能模块 | 核心功能 | 交互方式 |
|------|----------|----------|----------|
| 1 | **Header 顶栏** | 版本显示、导出报告、复制链接、实时时钟 | 按钮点击 |
| 2 | **全局筛选栏** | 时间、市场、品牌、城市、架构人员、门店搜索 | 下拉/输入 |
| 3 | **数据信息栏** | 更新日期、门店总数、均出餐时长 | 只读 |
| 4 | **大盘业绩概览** | 外卖门店总数、按市场/品牌分布饼图 | 鼠标悬停 |
| 5 | **顶部指标卡片** | 分平台 9 项核心指标 + 环比变化箭头 | 只读 |
| 6 | **渠道双栏面板** | 左：小程序热力图+趋势图+分段占比 | 鼠标悬停、时间联动 |
|  |  | 右：美团评分树+趋势图 |  |
| 7 | **管理排名** | 市场/品牌排名、大区排名、区经理排名、单店明细 | 点击下钻、列排序 |
| 8 | **告警面板** | 三类异常（出餐/店铺分/体验分）按大区分组 | 点击展开详情 |
| 9 | **标杆/预警门店** | 左侧：标杆门店（三项全达标） | 点击下钻、列排序 |
|  |  | 右侧：预警门店（任一项不达标） |  |
| 10 | **改善指南** | 按类别分组的改善建议 | 只读参考 |
| 11 | **更新日志** | 版本变更记录 | 只读 |

### 5.2 关键模块详解

#### 模块 A — 渠道双栏面板

| 功能 | 左侧：自有小程序 | 右侧：第三方平台 |
|------|------------------|-------------------|
| 核心可视化 | 出餐时间逐时热力图（8时-17时） | 店铺分六维评分树 |
| 辅助可视化 | 近7日出餐趋势折线图 + SLA参考线 | 体验分六维评分树 |
| 底部信息 | 出餐时长的四段占比条 | 近7日评分双轴趋势图 |

热力图色彩编码：

| 出餐范围 | 颜色 | 语义 |
|----------|------|------|
| ≤ 6 分钟 | 🟢 深绿 | 优质 |
| 6 - 12.75 分钟 | 🟢 浅绿 | 良好 |
| 12.75 - 15 分钟 | 🟡 黄色 | 预警 |
| > 15 分钟 | 🔴 红色 | 超时 |

#### 模块 B — 管理排名系统

四个排名表格，形成从粗到细的层级下钻：

```
市场/品牌排名（~7行）
  ├─ 点击「A市场」→ 下钻到该市场的所有门店
  │    └─ 点击某门店 → 门店详情抽屉（MP/MT双Tab）
  └─ 点击「X品牌」→ 下钻到该品牌的所有门店

大区经理排名（~10人）
  └─ 点击某大区经理 → 下钻到其辖区的所有门店

区经理排名（~30人）
  └─ 点击某区经理 → 下钻到其辖区的门店（按大店长分组）
```

**单店数据模块**：在全部门店粒度上的全景明细表，展示门店名、所属市场/城市、出餐时间、店铺分、体验分。异常门店整行标红加粗，方便快速扫视找出所有问题门店。

> 📷 **[配图建议 5]**：管理排名区域截图，标注"从市场排名（左列）→ 区经理排名（右列）→ 单店明细（左下），点击任意行即可下钻"。

#### 模块 C — 门店详情抽屉

点击任意门店，右侧滑出诊断抽屉，分为两个 Tab：

| 区块 | 自有小程序 Tab | 第三方平台 Tab |
|------|---------------|----------------|
| 诊断总结 | 自动判定 OK / WARN / BAD 状态 | 同上 |
| 核心指标 | 出餐时长、15分钟达标率 | 店铺分、体验分 |
| 时效分析 | 10小时逐时出餐热力图 | — |
| 趋势分析 | 近7日出餐趋势图 | 近7日评分双轴图 |
| 维度拆解 | — | 店铺分六维 + 体验分六维进度条 |

每个子维度进度条包含：当前值、目标线、达标/预警/未达标 三种状态色。

#### 模块 D — 告警与标杆系统

**标杆门店**：同时满足三项条件（出餐 ≤ 15分、店铺分 ≥ 95、体验分 ≥ 4.6），按综合评分排序。可为绩效表彰和最佳实践推广提供依据。

**预警门店**：任一项指标不达标即进入列表，按严重度排序。严重度计算公式同时考虑偏离程度和指标权重。

**告警面板**：按大区经理分组，直观展示"哪些大区有问题、分别有几家"。三个告警维度均有独立颜色编码。

## 六、筛选与联动逻辑

### 6.1 筛选维度

| 筛选维度 | 控件类型 | 联动行为 |
|----------|----------|----------|
| 时间范围 | 日期选择 + 快捷按钮 | 影响全局所有数据 |
| 市场 | 多选下拉 | 切换时重置城市、大区、区经理、大店长 |
| 品牌 | 多选下拉 | 同上 |
| 城市 | 多选下拉（联动市场） | 切换时重置架构筛选 |
| 大区经理 | 多选下拉 | 联动更新区经理、大店长下拉 |
| 区经理 | 多选下拉 | 联动更新大店长下拉 |
| 大店长 | 多选下拉 | 无下级联动 |
| 门店搜索 | 模糊匹配输入框 + 自动补全 | 直接定位到单店 |

### 6.2 级联刷新机制

筛选条件变更时，页面中所有依赖筛选的数据区域**同步刷新**：

- 顶部指标卡片重新计算
- 趋势图重新拉取对应时间范围数据
- 排名表格重新过滤排序
- 预警面板重新评估
- 标杆/预警门店列表重新筛选

### 6.3 筛选状态可视化

- 页面顶部的筛选提醒条实时展示当前激活的筛选条件
- 每个筛选标签旁带"×"按钮，可单独移除
- 时间范围快捷按钮（昨日 / 近7天 / 近30天 / 全部）一键切换

## 七、数据更新机制

### 7.1 自动化流程

```
每日 11:00（定时触发）
  │
  ├─ 1. 检查 D:\工作\workbuddy\ 下 5/6/7 月数据源文件
  │     （确保文件存在且为最新）
  │
  ├─ 2. 执行 gen_data_v4.py
  │     → 读取三个月份 Excel → 合并、去重、聚合
  │     → 输出 dashboard/data.json（~3MB）
  │     → 统计匹配率、门店数、异常记录
  │
  ├─ 3. 执行 embed_data.py
  │     → 将 data.json 嵌入到 dashboard/index.html
  │     → 复制到根目录 index.html
  │     → 更新版本号和构建时间戳
  │
  ├─ 4. 运行 verify_dashboard.py（可选）
  │     → 校验 data.json JSON 格式和必要字段
  │     → 校验 index.html 是否包含嵌入数据标记
  │     → 退出码 0=通过 / 1=警告 / 2=错误
  │
  └─ 5. git push github master
        → GitHub Pages 自动部署
        → 团队成员访问 https://<username>.github.io/takeout-dashboard/
```

### 7.2 异常处理

| 异常场景 | 处理策略 |
|----------|----------|
| 数据源文件缺失 | 跳过缺失月份，仅用已有数据生成（日志记录） |
| 脚本执行失败 | 保留上一次成功的 index.html，发送告警通知 |
| Git 推送冲突 | 强制覆盖远程（数据看板为只读消费端，不怕冲突） |
| GitHub Pages 缓存 | 每次部署更新版本号参数，强制浏览器拉取最新版本 |
| 月份轮转（如新增 8 月） | 修改脚本中的 OLD/MID/NEW 路径变量，确认新文件就绪后执行 |

### 7.3 数据完整性验证

每次生成 data.json 后记录：

- 美团门店数、小程序门店数、双平台重叠门店数
- 架构匹配率（目标 ≥ 95%）
- 各市场、各品牌门店分布
- 预警门店数量

异常数值（如匹配率骤降、某市场门店数为 0）触发人工确认。

## 九、技术实现要点

### 9.1 单文件 HTML 架构

整个看板为单个 `index.html` 文件（~2.5MB），包含：
- HTML 结构（~800 行）
- CSS 样式（~400 行）
- JavaScript 业务逻辑（~3300 行）
- 内联 Chart.js 4.x 图表库
- 内联 chartjs-plugin-datalabels 插件
- 嵌入式 JSON 数据（~1.5MB）

**优势**：
- 零依赖构建：无需 npm/webpack/vite，浏览器直接打开可用
- 部署极简：一个文件推到 GitHub Pages 即上线
- 离线可用：数据嵌入在 HTML 中，本地打开无需网络

### 9.2 下钻与状态管理

- 右侧滑出抽屉（520px 宽），支持返回栈（`drawerBackStack`）
- 每层下钻保存上一级状态（筛选条件、排序方式），点击返回可逐层回退
- 门店详情抽屉分 MP/MT 两个 Tab，根据门店实际平台动态显示

### 9.3 筛选性能优化

- 使用 `filterStores()` 在前端内存中过滤（数据量 ~500 门店，过滤 < 10ms）
- 趋势图仅渲染可见范围数据，避免大数组的 Chart.js 重绘
- 排名表格使用字符串拼接后一次性 `innerHTML` 渲染，避免逐行 DOM 操作

### 9.4 兼容性

- 支持 Chrome / Edge / Safari 主流浏览器
- 移动端响应式适配：抽屉全屏、排名表横向滚动
- 颜色方案遵循中国资本市场习惯（红涨绿跌惯例不适用，但在设计上红=异常/绿=正常）
